# Pivot Network 架构说明

更新时间：`2026-04-11`

## 1. 当前文档定位

这份文档描述当前 repo 的项目级架构现状，以及 seller 到 buyer 的正式闭环目标。

当前项目级权威阅读顺序建议是：

1. `PROJECT.md`
2. `Seller_Client/docs/current-seller-onboarding-flow-cn.md`
3. `Plantform_Backend/README.md`
4. `Buyer_Client/docs/current-buyer-purchase-flow-cn.md`
5. `Docker_Swarm/Docker_Swarm_Adapter/README.md`
6. 本文

## 2. 当前架构主结论

当前正式架构不是单一 seller 链，而是两条相连的链：

1. seller 接入与商品化链
   - `Seller_Client -> Plantform_Backend -> Docker_Swarm_Adapter -> Docker Swarm`
2. buyer 交易与会话链
   - `Buyer_Client -> Plantform_Backend -> Docker_Swarm_Adapter / WireGuard / Managed Runtime`

并且有 4 条硬约束：

1. seller 本地执行 `docker swarm join`
2. seller client 不直连 adapter
3. `manager_acceptance` 由 backend 通过 adapter 真相生成
4. seller join 的完成标准是 manager-side task execution，而不是本地 `docker info` 自报

buyer 侧还需要再固定 3 条硬约束：

1. buyer 买的是 `RuntimeSession`，不是 seller 裸 IP
2. `activate order = issue grant`
3. `redeem grant + wireguard public key = create runtime session`

## 4. 当前 seller 侧架构

### 4.1 Seller Client

当前正式入口已经收束到：

- `bootstrap/windows/install_and_check_seller_client.ps1`
- `bootstrap/windows/start_seller_client.ps1`

本地 Web 壳目前固定为 5 个工作区：

- `环境`
- `网络 / WireGuard`
- `Docker / Swarm`
- `接入会话`
- `AI 助手`

自然语言主线默认走 MCP 编排，不再走旧的本地裸 workflow 入口。

### 4.2 当前 seller 完成标准

根据当前 `Seller_Client` 代码，卖家接入最终成功标准已经收束成：

- manager 侧看到 worker `Ready`
- manager 侧确认该 worker 上存在可执行或运行中的 task

这意味着：

- `WireGuard connectivity`
- 本地 `docker info`
- 本地 `LocalNodeState=active`

都只能当作中间证据，不能单独算成功。

## 5. 当前 backend 架构职责

`Plantform_Backend` 当前是业务真相面。

它负责：

- 账号与登录态
- `JoinSession`
- Phase 1 probe / `join-complete` 写入
- `manager_acceptance`
- `corrections`
- `re-verify`
- `authoritative-effective-target`
- `minimum-tcp-validation`
- capability assessment
- verified-node 自动商品化
- `offer / order / access_grant`

它不负责：

- 远程代执行 seller 机器上的 join
- 直接充当 Swarm 控制器

当前 backend 已经补上：

- `verified onboarding session -> assessment -> offer upsert/downlist`

但还没有补上：

- `grant -> runtime session -> task execution`

## 6. 当前 adapter / swarm 架构职责

`Docker_Swarm_Adapter` 当前是基础设施适配层。

它负责：

- `join-material`
- 节点 inspect / search / claim
- runtime bundle / wireguard 租约

它不负责：

- 保存 `JoinSession`
- 保存 `effective_target`
- 定义 seller onboarding 完成标准

`Docker_Swarm/` 则是 adapter 的运行环境和 Swarm manager 运维目录。

## 7. 当前服务端真相模型

backend 当前维护两层服务端真相：

### 7.1 Raw manager truth

- `manager_acceptance`
- 来自 backend 对 adapter 的 `inspect / claim / inspect`

### 7.2 Effective target truth

- `effective_target_addr`
- `effective_target_source`
- `truth_authority`

当前优先级：

1. `manager_matched`
2. `backend_correction`
3. `operator_override`

`minimum_tcp_validation` 仍然存在，但它现在更适合被理解为：

- correction lane 的附加证据
- 后续 buyer/connect 语义输入

而不是 seller join 的完成标准本身。

## 8. 当前 buyer 相关位置

当前 repo 已经有正式 `Buyer_Client/` 目录，但 buyer 侧当前仍以文档规格为主。

当前 buyer 侧已经固定：

- 主产品语义是 `云主机会话`
- 正式 shell 入口在本地 `Buyer_Client`
- `Order / AccessGrant / RuntimeSession / TaskExecution` 的分层关系

当前 buyer 侧尚未完成：

- `grant-first` 正式接口链
- `RuntimeSession` 实现
- WireGuard / shell / workspace / task 数据平面

因此当前项目不能再说“没有 Buyer_Client”，但也不能误判为“buyer 已实现完成”。

## 9. 当前 seller 到 buyer 的正式闭环目标

项目级闭环目标固定为：

1. seller 节点接入并被 backend 验收
2. backend 自动测算并上架真实 offer
3. buyer 浏览真实 offer 并下单
4. backend 激活订单并签发 grant
5. buyer client 兑换 grant 创建 `RuntimeSession`
6. adapter 创建 runtime bundle
7. buyer client 通过 WireGuard 进入 shell 并执行任务

## 10. 当前最重要的架构边界

- 不把 `81.70.52.75:2377` 写回 seller join target
- 不把 seller 本地观测当成 manager 最终真相
- 不让 seller client 绕过 backend 直接调用 adapter
- 不把 `minimum_tcp_validation` 重新提升成 seller join 的唯一完成标准
- 不把 buyer 产品语义退回成“直接拿 seller target”
- 不让 buyer client 绕过 backend 直接创建 runtime bundle 或 WireGuard lease

## 11. 当前阶段节奏

项目级阶段节奏固定为：

- `phase3` 末期
  - 继续验证 seller 节点商品化与平台上架逻辑
- `phase4`
  - Buyer_Client 设计与实施规格
- `phase5`
  - 卖家、平台、买家联调闭环

因此当前更准确的表述是：

- seller onboarding 与 verified-node 商品化已进入验证/稳定阶段
- buyer 进入正式设计阶段
- 端到端闭环联调是下一阶段目标
